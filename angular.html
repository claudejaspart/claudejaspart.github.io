Sending GET requests




We want to get all the posts we sent :

this.http.get('https://angular-api-testing.firebaseio.com/posts.json').subscribe(
    (posts) =>
    {
    console.log(posts);
    }
);

We want to add that json data into an array. 

Using RxJS operators to transform data response :

We will be using the map operator. We will loop through all the keys and create a new array using object 
destructuring using the typescript spread operator :

import { map } from 'rxjs/operators';

this.http.get<{[key:string]: Post}>('https://angular-api-testing.firebaseio.com/posts.json')
.pipe(
  map(responseData  =>
    {
      const postsArray = [];
      for (const key in responseData)
      {
        if(responseData.hasOwnProperty(key))
          postsArray.push({...responseData[key], id: key});
      }
      return postsArray;
    }
  )
)
.subscribe(
  (posts) =>
  {
    this.loadedPosts.push(posts);
    console.log(this.loadedPosts[0]);
  }
);

Where post is an interface : 

export interface Post
{
    title : string;
    content: string;
    id? : string;
}

Displaying the posts :

<p *ngIf="loadedPosts.length < 1">No posts available!</p>
<ul class="list-group" *ngIf="loadedPosts.length > 0">
    <li class="list-group-item" *ngFor="let post of loadedPosts">
        <h3>{{ post.title }}</h3>
        <p>{{ post.content }}</p>
    </li>
</ul>

Logic for a loader :

Create a variable called isFetching set to false;

Before calling the GET request, set it to true. When we receive the data in the subscribe, set it to false.
Use ngIf in the Html template to display it while isFetching is true.

Creating custom pipes




As an example, we will create a pipe that shortens a word.

CLI way :

ng g p pipe_name

Manual way :

1. Create a file name shorten.pipe.ts in the root directory

2. Code in that file :

@Pipe({
    name:'shorten',
    pure : false        // by default is true, recalculates the pipe each time the data changes
                        // might create performance behavior with a long list
})
import { Pipe, PipeTransform } from "@angular/core";

export class ShortenPipe implements PipeTransform
{
    transform(value: any) 
    {
        return value.substring(0,4) + "...";
    }    
}

3. In the appModule.ts :

import { ShortenPipe } from './shorten.pipe'

@NgModule(

    ...

It is ready to be used.

Reactive - Creating a custom async validator




We want to create a validator that will take some time to process, like a response from a server.

Instead of returning an object { string: boolean}, we will return a promise or an observervable of any type :

// step 1 : creating our custom validator
forbiddenEmails(control : FormControl) : Promise<any> | Observable<any>
{
    const promise = new Promise<any>((resolve, reject)=>{
    setTimeout(()=>{
        if (control.value === 'test@test.com')
        resolve({'emailIsForbidden': true});
        else
        resolve(null);
    }, 1500);
    });

    return promise;
}

// step 2 : adding our validator in the signupForm :
// note that the 3rd argument is for async validators
// no need for the bind(this) since we are not using the 'this' keyword in the validator
'email' : new FormControl(null, [Validators.required, Validators.email], this.forbiddenEmails)

We can inspect the element and we see that when the async validator is called, we have the ng-pending class that is added.


















Reactive - Custom validators




The built-in validators should cover most of the use-cases.
For example, we could create a validator to check for forbidden user names.

!!! A validator is successful when we return a null object, otherwise, the validator has failed. 

Code example in ts file:

// step 1 : create our invalid names :
forbiddenUsernames = ['Chris','Anna'];

// step 2 create the validator : 
// example of return value :
// return {'anna': true}
forbiddenNames(control : FormControl) : { [s:string]: boolean }
{
  if ( this.forbiddenUsernames.includes(control.value) )
    return {'nameIsForbidden' : true};
  else
    return null;
}

// step 3 : add our validator in the signupForm :
// note the use of .bind(this) to make sure we bind it to our current class
this.signupForm = new FormGroup(
    {
      'userData' : new FormGroup(
        {
            'username' : new FormControl(null, [Validators.required, this.forbiddenNames.bind(this)]),  <===
            'email' : new FormControl(null, [Validators.required, Validators.email]),
            'gender' : new FormControl('male', Validators.required)
        }
      ),
      'hobbies' : new FormArray([])
    }
);

The error code will be located in the selected control object, in the errors array.
We can setup a custom error code with :

*ngIf="signupForm.get('userData.username').errors['nameIsForbidden']"

Subjects




A subject is an "active" observer as we can trigger the return event from code, where as we have to wait for the response 
of a regular observable (passive). It works basically the same as an event emitter.

import { Subject } from 'rxjs';

// code in the user.service.ts :
activatedEmitter = new Subject<ValueType>()

// Calling the subject to trigger a change :
this.userService.activatedEmitter.next(value_to_change);

// Receiving the data :
this.userService.activateEmitter.subscribe( (data) => { myCode });

!!! Do not use EventEmitter for direct 'in code' changes between components !!!

You need to destroy the subject when exiting the component with :

this.userService.unsubscribe();


















Operators




Operators are the magic features of the rxjs library : right before the data arrives to the observers, operators
process the data. There are a ton of available operators.

You call an operator with the pipe() method :

import { operatorName (ex: map) } from 'rxjs/operators';

myObservable.pipe(map((data)=>{return doSomething})).subscribe((data)=>{do something})    

The pipe() method takes an unlimited amount of arguments. 
















Creating a custom observable




First, import the Observable package from rxjs :

import { Observable } from 'rxjs';

Create an observable :

const customObs = Observable.create(
    (observer)=>
    {
        // 3 possible functions we can use :
        observer.next(count);                       // next emitted value, for example data
        observer.error(new Error(message));         // throw an error
        observer.complete();                        // observable is finished
    }
);

subscribe to the observable :

customObs.subscribe((data)=>{ your code }, (error)=>{ your code }, ()=>{ obs completed });

!!! An error cancels the observable, it never completes !!!

You very very rarely create your own observables.
















Using a built-in observable




The rxjs package gives us different ways to create a new observable. One of the easiest way is to use "interval", a built-in 
function :

import { interval } from rxjs;

It will fire an event every x milliseconds :

this.ObsInterval = interval(1000).subscribe( count =>
{
    console.log(count);
})

Some observable keep on going (like interval), other just emit an event and are done (like http request). It is very
important to manage the observable when we quit the component to avoid memory leak. We can use the ngOnDestroy hook for example :

this.ObsInterval.unsubscribe();

Angular will take care of cleaning all observables that are built-in.

Note : A variable with a $ suffix is used to indicate that the variable is an Observable. Ex : user$


















Introduction




An observable can be seen a data source : user input, HTTP requests, database access ...
We have an observable and an observer (your code). In between, we have a stream (a timeline) where we can have multiple events
emitted by the observable.
We receive three ways of handling data packages :
_ Handle Data 
_ Handle Error 
_ Handle Completion

An observable might never complete.




























Passing dynamic data




We will use a resolver that will execute some code before rendering/displaying the component.
We need to create a service which implements the Resolve interface.

import { Resolve, ActivatedRouteSnapshot, RouterStateSnapshot } from '@angular/router'

interface Server
{
    id:number;
    name:string;
    status:string;
}

export class ServerResolver implements Resolve<Server>
{
    resolve(route:ActivatedRouteSnapshot, state: RouterStateSnapshot)
    : Observable<Server> | Promise<Server> | Server
    {
        return this.serverService.getServer(+route.params['id']);
    }
}

We need to add it to the service provider in AppModule.ts and use it in the router paths :

{ path : 'users/:id/:name',component:UserComponent, resolve:{server:ServerResolver}},

We can access the data with the 'data' baggage :

this.route.data.subscribe(
    (data : Data ) =>
    {
        // this is the name of the object used in the resolver just above :
        // resolve:{server : ServerResolver}
        this.server = data['server'];
    }
)

CanActivate Guard




We want to run some code before some component is loaded. For example, to check if a user has access to this component.

CanActivate

This method will be called by Angular before a route is loaded.

CanActivate can return both asynchronously (observable or promise) or synchronously.

import { CanActivate, ActivatedRouteSnapshot, RouterStateSnapshot } from "@angular/router";
import { Observable } from "rxjs";

export class AuthGuard implements CanActivate
{

    canActivate( route:ActivatedRouteSnapshot, state: RouterStateSnapshot)
                : Observable | Promise | boolean 
    {

    }

}

Full example :

We create here a AuthGuard Service :

import { CanActivate, ActivatedRouteSnapshot, RouterStateSnapshot } from "@angular/router";
import { Observable } from "rxjs";
import { Injectable } from "@angular/core";
import { AuthService } from './auth.service';
import { Router } from '@angular/router';

@Injectable()
export class AuthGuard implements CanActivate
{
    constructor(private authService : AuthService, private router: Router){};


    canActivate( route:ActivatedRouteSnapshot, state: RouterStateSnapshot)
                : Observable | Promise | boolean 
    {
        return this.authService.isAuthenticated().then(
            (authenticated : boolean) =>
            {
                if (authenticated)
                {
                    console.log("logged in");
                    return true;
                }
                else
                {
                    console.log("not logged in");
                    this.router.navigate(['/']);
                }
            }
        );
    }
}

In the app-routing.module.ts, we select the routes that need the AuthGuard service :

const appRoutes: Routes = 
[
    { path : '',                      component: HomeComponent    },			
    { path : 'users',                 component: UsersComponent   },
    { path : 'users/:id/:name',       component:UserComponent     },
    { path : 'servers',               canActivate:[AuthGuard],    <=====   
                                      component: ServersComponent , 
                                      children : 
                                        [
                                            { path : ':id',         component : ServerComponent },
                                            { path : ':id/edit',    component : EditServerComponent }
                                        ]},
];  

Then, in the appModule.ts, we need to add the services to the providers, without forgetting the imports :

providers: [ServersService, AuthService, AuthGuard],

Protecting child routes :
We can use the method CanActivateChild, which has the same form as CanActivate. Thus, we can just call the CanActivate method
inside CanActivateChild.

We can protect either all the routes (parent and child), or just a parent or the children.

Redirection and wildcard routes




Redirection :

We use the redirectTo property in the appModule list of paths :

const appRoutes: Routes = 
[
    { path : 'not-found',   component: PageNotFoundComponent    },			
    { path : 'something',   redirectTo: '/not-found'            }
];

wildcard routes :

const appRoutes: Routes = 
[
    { path : 'not-found',   component: PageNotFoundComponent    },			
    { path : '**',   redirectTo: '/not-found'            }
];

** means catch all routes we don't know. This generic route must be the last one in the list.

Important: Redirection Path Matching

In our example, we didn't encounter any issues when we tried to redirect the user. But that's not always the case when 
adding redirections.
By default, Angular matches paths by prefix. That means, that the following route will match both /recipes and just / : 

{ path: '', redirectTo: '/somewhere-else' } 

Actually, Angular will give you an error here, because that's a common gotcha: This route will now ALWAYS redirect you! Why?
Since the default matching strategy is "prefix" , Angular checks if the path you entered in the URL does start with the 
path specified in the route. Of course every path starts with ''  (Important: That's no whitespace, it's simply "nothing").
To fix this behavior, you need to change the matching strategy to "full" :

{ path: '', redirectTo: '/somewhere-else', pathMatch: 'full' } 

Now, you only get redirected, if the full path is ''  (so only if you got NO other content in your path in this example).

Fetching route parameters reactively

We have this.route.params which is an observable. Observables are a feature from a 3rd party package (not from Angular),
that lets us work with asynchronous tasks.
An observable is an easy way to subscribe to an event that might happen in the future, and execute some code when it happens 
without having to wait for it now. 
This.route.params.subscribe() has 3 function parameters that we can pass. The first one is the most important :
The first function is executed when new data is fired by the observable:

    (params : Params) => 
    {
        this.user.id = params['id'];
        this.user.name = params['name'];
    }
);
Angular does cleanup of the observable when the component is destroyed. Theoretically, you might want to implement
the onDestroy lifecycle (especially when working with your own observables) :

// imports
import { Subscription } from 'rxjs/Subscription';

// constructor
paramsSubscription: Subscription;

// core code
this.subscription = this.route.params.subscribe(
    (params : Params) => 
    {
        this.user.id = params['id'];
        this.user.name = params['name'];
    }
);

// kill subscription
ngOnDestroy()
{
    this.subscription.unsubscribe();
}

Understanding the hierarchical Injector




The Angular dependency injector is actually a hierarchical injector : if we provide a service in some place of our app, 
lets say a component, then angular knows how to create an instance of this service for that component and all of its childs. 
They will all receive the same instance of that service.  
The highest level is the AppModule : the service is available application-wide : all components and services (services 
that use services).
Then we have AppComponent : Same instance of the service is available for all components, but not for other services.
Lowest level : component without children. This will overwrite the instance of the service if its defined on a higher level.

!!!!  To use an existing service, just don't provide it in the providers. !!!!

We still need to use the constructor and the import.

The component lifecycle




Once a component is initiated, angular goes through different phases in the creation process.

In order :

ngOnChanges             

    May be executed multiple times. Called whenever a bound input property changes (properties decorated with @Input).

ngOnInit

ngDoCheck

ngAfterViewChecked      

    Called every time the view and child views have been checked.

ngOnDestroy

export class ServerComponent
{
    serverId: number = 10;
    serverStatus = 'offline';
}
In the component's template :

    
<p>Server with {{serverId}} is {{serverStatus}}</p>

We can also use a method :

serverId: number = 10;
serverStatus = 'offline';

getServerStatus()
{
    return this.serverStatus;
}

      
Case 2 : Property binding

In the ts, we add a variable. This variable is then used a property value in the html using the syntax : 

[property]="variable_name"

Example :

<button class="btn btn-primary" [disabled]="!allowNewServer">Add server</button>

    
Case 3 : Event binding
    
Start event names functions with "on" : onCreateServer()
    
Call a function on a button click :

(click)="onCreateServer()"

To catch in the TS what the user input is : 

<input type="text" class="form-control" (input)="onUpdateServerName($event)">

The $event is super important. This keyword gives us access to event data such as mouse position, 
etc...

We can retrieve the text with : 

onUpdateServerName(event:Event)
{
    this.serverName = (event.target as HTMLInputElement).value;
}

   
Case 4 : Two-way-binding

!!!! FormsModule is required for 2-way-binding !!!!!
    
This is done by :

1. adding the FormsModule to the imports[]  array in the AppModule.

2. You then also need to add the import from @angular/forms  in the app.module.ts file:

        import { FormsModule } from '@angular/forms'; 
    
With 2Waybinding, we combine property and event binding :

<input type="text" class="form-control" [(NgModel)]="serverName">

   


                

            

            








            
            

                
Components

                

                

                    
An angular app is composed of many components we create.

There is a root component in which we nest multiple components.

Adding a component to the root component :

Create the component folder as a subfolder of app/.

Naming convention : componentName.component.html

Ex : for server component :  server.component.html

Lets edit our server.component.ts :


    // to use the Component decorator
    import { Component } from '@angular/core';


    // @Component is a decorator that specifies that ServerComponent
    // is a special type of class
    @Component(
    // component configuration object
    {
        selector: 'app-server',                 /*  component tag name */
        templateUrl: './server.component.html'  /*  html file */
    })
    export class ServerComponent
    {

    }

The component is ready to use but we need to add it in the module file.
An angular app can have multiple modules (aka packages).

To add a component to a module, we need to modify the app.module.ts :

    1. Import it : import { ServerComponent } from './server/server.component';

    2. Add it in the declarations array


The imports array beneath the declarations let us add other tiers modules.

Creating a component in the CLI :

> ng generate component component-name
> ng g c path/component-name
> ng g c component-name --skipTests true

HTML templates

In the .ts file of the component, there has to be either templateUrl or template.

templateUrl => external html file
template    => inline html

We can use backticks to split into multiple lines and use js string interpolation.

Styling the components

The style of a template is uniquely defined in its own .css component file.

Using the attribute selector [] to select a component

In the server.component.ts we change the name of the selector with added brackets :

    selector: 'app-server' => selector: '[app-server]'

This lets us insert the component with a html tag :

    <div app-server></div>

Using the class selector . to select a component

In the server.component.ts we change the name of the selector with added brackets

    selector: 'app-server' => selector: '.app-server'

This lets us insert the component with a html tag :

    <div class="app-server"></div>

Note : selecting by id instead of a class is not implemented.

Directives

                

                

                    
Directives are instructions in the DOM.

Two types of directives :

Attribute directives : Only alter the properties of the DOM, ie the BackgroundColor

Structural directives : Alters the DOM by destroying and adding elements in it

Note : We cannot have more than one structural directive on the same element. 

Example :

    <p appHighlight>Receives a highlight !</p>

    @Directive(
    {
        selector:'[appHighlight]'
    })
    export class HightlightDirective
    {	}

You can generate a directive with the CLI :

    > ng generate directive highlight

The CLI creates src/app/highlight.directive.ts, a corresponding test file src/app/highlight.directive.spec.ts, 
and declares the directive class in the root AppModule.   

After editing the hightlight.ts (note the use of ElementRef):

import { Directive, ElementRef } from '@angular/core';

@Directive({
  selector: '[appHighlight]'
})
export class HighlightDirective 
{
    constructor(el: ElementRef) 
    {
       el.nativeElement.style.backgroundColor = 'yellow';
    }
}

ngModel Example

                

                

                    
Component.ts :


    @Component(
    {
        selector: 'app-root',
        templateUrl: './app.component.html',
        styleUrls: ['./app.component.css']
    })
    export class AppComponent 
    {
        name = 'Claude';
    }


Component.html : 

    input type="text" [(ngModel)]="name"
    <p>{{ name }}</p>

When you try to run the app and do F12 :

Can't bind to 'ngModel' since it isn't a known property of 'input'.

We need to import ngModel in app.module.ts :

    import { BrowserModule } from '@angular/platform-browser';
    import { NgModule } from '@angular/core';
    import { FormsModule } from '@angular/forms';  < here

    import { AppComponent } from './app.component';

    @NgModule({
        declarations: [
        AppComponent
        ],
        imports: [
        BrowserModule,
        FormsModule 	< here
        ],
        providers: [],
        bootstrap: [AppComponent]
    })
    export class AppModule { }

This will cause the input text to start with the name value and whenever we change the input text, 
then the name display will change too.
                    

                

            

            




















            
            

                
ngIf

                

                

                    
ngIf is a structural directive : it changes the structure of our DOM.
Structural directives start with a star : *

Simple if :

    <p *ngIf="boolean_variable">
        ... do something ...
    </p>

if else :

We need to use a template variable to reference it in the else clause.

<p *ngIf="serverCreated, else noServer">
    Server was created, server name is {{ serverName }}
</p>

<ng-template #noServer>
    <p>No server was created.</p>
</ng-template>



Complete if then else :

    <ng-container *ngIf="isLoggedIn; then loggedIn; else loggedOut">
    </ng-container>


                    
                

            

            
















            
            

                
ngStyle

                

                

                    
Attribute directives don't add or remove elements. They only change the element they were placed on.

ngStyle dynamically changes the style.

We use brackets to bind this directive to some property :

    <p [ngStyle]="{backgroundColor: 'red'}">Server status</p>

But we want it to be dynamic, so :

    <p [ngStyle]="{color: getColor()}">Server status</p>

with :

    function getColor()
    {
        return this.serverStatus === 'online' ? 'green' : 'red';
    }


                    

                

            

            





            






        
        

            
ngClass

            

            

                
This attribute directive lets us add or remove classes when a condition is true :

[ngClass]="{class_name="boolean_variable"}"
